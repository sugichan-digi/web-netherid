## 1. セキュリティ

### 🔴 S-01 お知らせ本文を無害化せず HTML として挿入している（XSS リスク）

**ファイル**: `frontend/notification/script.js:283`

```js
// ❌ 危険: API レスポンスの content をそのまま innerHTML に挿入している
$('#notif-modal-body').html(item.body);
```

コード内コメントに「HTML を含む場合があるため innerHTML として挿入する」とあるが、DB に不正なスクリプトが混入した場合や API が改ざんされた場合に XSS が成立する。
本文をリッチテキスト（HTML）として表示する要件があるなら DOMPurify 等でサニタイズすること。要件がないなら `text()` に切り替えること。

```js
// ✅ 要件次第で選択する
$('#notif-modal-body').text(item.body);          // プレーンテキストのみ許容する場合
// または
$('#notif-modal-body').html(DOMPurify.sanitize(item.body)); // リッチテキストを許容する場合
```

$cookieHeader = collect($request->cookies->all())  // ← 完全に重複

### 🟠 B-03 `KratosOptionalAuth.php` で `env()` を `config()` のフォールバックに直接使っている

**ファイル**: `backend/app/Http/Middleware/KratosOptionalAuth.php:38`

```php
// ❌ 本番では php artisan config:cache 後に env() が機能しなくなる
$kratosUrl = rtrim(config('app.kratos_public_url', env('KRATOS_PUBLIC_URL', 'http://localhost:4433')), '/');
```

Laravel のベストプラクティスでは、コントローラー・ミドルウェア等のアプリケーションコードから `env()` を直接呼ぶことは禁止されている。`config:cache` 実行後は `.env` が読み込まれなくなるため、本番環境で値が `null` になるリスクがある。

```php
// ✅ config/app.php に追記してから参照する
$kratosUrl = rtrim(config('app.kratos_public_url'), '/');
```

一方 `KratosAuth.php:45` では `config('app.kratos_public_url')` と正しく書かれており、2ファイル間で書き方が統一されていない。

### 🟠 B-04 `NotificationService` / `ServiceService` のレスポンスが生配列でルール違反

**ファイル**: `backend/app/Services/NotificationService.php:23-33` / `backend/app/Services/ServiceService.php:31-40`

CLAUDE.md の「**値オブジェクト（readonly class）を活用して構造化（配列の生渡し禁止）**」というルールに違反している。
`InquiryService` では `CreateInquiryInput` という `readonly class` が使われているが、`NotificationService` と `ServiceService` では連想配列を直接返している。

```php
// ❌ 配列の生渡し（ルール違反）
return $services->map(fn ($service) => [
    'id'          => $service->id,
    'name'        => $service->name,
    // ...
])->values()->all();
```

```php
// ✅ readonly class の値オブジェクトで構造化する（例）
// App\Data\ServiceDto.php
readonly class ServiceDto {
    public function __construct(
        public int    $id,
        public string $name,
        // ...
    ) {}
}
```

## 3. フロントエンド コーディングルール違反

### 🟠 F-01 `design-system/index.html` が相対パスで CSS を読み込んでいる（ルール違反）

**ファイル**: `frontend/design-system/index.html:9`

```html
<!-- ❌ 相対パス（ルール違反） -->
<link rel="stylesheet" href="style.css">

<!-- ✅ ルート相対パス -->
<link rel="stylesheet" href="/design-system/style.css">
```

CLAUDE.md フロントエンドルール「**パス指定: リソース読み込みは必ずルート相対パス（絶対パス: `/` から始まるパス）を使用する**」に違反している。

---

### 🟠 F-02 `common.js` で開発モードフラグが本番ハードコードされている

**ファイル**: `frontend/common.js:6`

```js
var ENV = 'develop';
```

`ENV === 'develop'` の条件でコンソールログが出力される実装であれば、本番環境でも全デバッグログが出力される。ビルドツールを使わない静的HTMLの構成であっても、本番デプロイ時に手動で `'production'` に書き換えるフローが必要。現状その手順が存在しない。

---

### 🟠 F-03 `common.js` で接続先URLがハードコードされている

**ファイル**: `frontend/common.js:10-12`

```js
var KRATOS_BASE = PROTOCOL + '//kratos.netherid-frontend.test:4433';
var API_BASE    = PROTOCOL + '//netherid-frontend.test:8000';
```

開発環境のホスト名がソースに直書きされており、本番デプロイ時に必ずファイルを書き換える必要がある。書き換え忘れが起きるリスクが高く、また Git 管理の観点でも開発・本番の差異がコード上に残る。

---

### 🟡 F-04 `common.js` でグローバル変数が大量に宣言されている

**ファイル**: `frontend/common.js`

`ENV`, `PROTOCOL`, `KRATOS_BASE`, `API_BASE` などが全て `var` でグローバルスコープに定義されている。ビルドツール不使用の制約上やむを得ない面もあるが、別スクリプトが同名変数を宣言した場合に上書きされるリスクがある。`window.NetherID = { ... }` のように名前空間オブジェクトにまとめることで衝突リスクを低減できる。

## 4. 設計上の懸念

### 🟠 D-01 クラス名 `ServiceService` が冗長で意図が不明確

**ファイル**: `backend/app/Services/ServiceService.php`

サービス層のクラスが「サービスに関するサービス」という冗長な命名になっている。`services` テーブルは「連携サービスマスタ」であるため、`LinkedServiceService` や `MasterServiceService` など意図が明確な名前が望ましい。

---

### 🟡 D-02 マイグレーションに PHPDoc クラスコメントがない

**ファイル**: `backend/database/migrations/2026_05_10_000001_add_service_id_to_notifications_table.php`

```php
// ❌ クラスレベルの PHPDoc なし
return new class extends Migration
{
    public function up(): void { ... }
```

CLAUDE.md の「**PHPDoc コメントは日本語・体言止めで全クラス・全メソッドに付ける**」というルールに照らすと、無名クラスであってもマイグレーション全体の説明コメントを付けることが望ましい。`up()` / `down()` は実装済みだがクラスコメントがない。
